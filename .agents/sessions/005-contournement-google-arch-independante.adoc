== Objectif

Valider le run end-to-end Colab → content-i18n/ → push office.git (CP-1d).
Le blocage Google "Ce navigateur ou cette application ne sont peut-être pas sécurisés"
empêchait toute automatisation Playwright depuis la session 001.

== Ce qui a été fait

=== 1. Contournement blocage Google anti-automation

*Problème* : Playwright `launchPersistentContext` ajoute `--enable-automation` que Google
détecte et bloque sur accounts.google.com/ServiceLogin.

*Solution* : Lancer Chromium snap en subprocess normal (ProcessBuilder), puis Playwright
se connecte via `connectOverCDP("http://localhost:9222")`. Google voit un navigateur normal.

*Détails* :
- Chromium snap : `/snap/chromium/current/usr/lib/chromium-browser/chrome` (v149)
- Flags : `--disable-blink-features=AutomationControlled`, `--no-first-run`,
  `--no-default-browser-check`
- Profil : `--user-data-dir=/home/cheroliv/snap/chromium/common/chromium` +
  `--profile-directory=Profile 2` (authentifié Google Colab)
- CDP : `--remote-debugging-port=9222 --remote-debugging-address=127.0.0.1`
- Attente CDP ready : polling HTTP sur `localhost:9222/json/version` (30s max)

=== 2. Run all via raccourci clavier Ctrl+F9

*Problème* : Les selectors UI Colab (`colab-runall-button`, `[label*="Run all"]`,
`getByText("Runtime")`) échouent à cause du shadow DOM Polymer de Colab.

*Solution* : `page.keyboard().press("Control+F9")` = raccourci Colab "Run all".
Fallbacks UI conservés au cas où le raccourci échoue.

=== 3. Polling git remplace polling DOM

*Problème* : `page.textContent("colab-terminal, .output_text, pre")` ne trouve jamais
`TRANSLATION_DONE=ok` dans le shadow DOM Polymer de Colab.

*Solution* : Polling `git ls-remote https://github.com/cheroliv/office.git refs/heads/main`
toutes les 30s pendant 30 min. Succès = HEAD sha change. Fallback DOM conservé
si `siteRepoUrl` non configuré.

=== 4. Architecture plugin indépendant (pattern article 0124)

*Problème* : `include("container-provision-plugin")` dans settings.gradle.kts racine =
monorepo classique (Option 2 article 0124). Pas de task `publishToMavenLocal` sur le
sous-module. Non conforme au pattern trouvé.

*Solution* :
- `container-provision-plugin/settings.gradle.kts` créé (build autonome)
- `container-provision-plugin/gradlew` + `gradle/wrapper/*` créés (wrapper autonome)
- `include("container-provision-plugin")` retiré de la racine
- `maven-publish` ajouté au build.gradle.kts du plugin (Gradle 9.x ne l'applique plus
  automatiquement avec java-gradle-plugin)
- `kotlin("jvm") version "2.3.20"` (version explicite, pas via catalogue)
- `publishToMavenLocal` maintenant fonctionnel depuis `container-provision-plugin/`

== Décisions techniques

. *Subprocess > LaunchPersistentContext* : Google détecte Playwright via les flags
  d'automation. Lancer Chromium en subprocess normal + connectOverCDP contourne le
  blocage. C'est la technique utilisée par les outils de scraping production.
. *Ctrl+F9 > UI selectors* : Le shadow DOM Polymer de Colab rend les selectors UI
  fragiles. Le raccourci clavier est fiable et documenté.
. *Polling git > Polling DOM* : Le DOM de Colab est trop complexe pour un polling
  fiable. Git est la source de vérité du livrable — c'est aussi plus résilient
  (si une cellule plante, on le voit via l'absence de commit).
. *Plugin indépendant > include()* : Conformité au pattern article 0124. Le sous-module
  est un build Gradle autonome avec son propre wrapper. La racine consomme le plugin
  via mavenLocal comme un client externe.
